# UI Workflow

UI 태스크는 **3개 에이전트가 순차적으로 관여**하는 파이프라인이다.

```
gemini-cli (디자인/리서치) → codex (구현) → claude-code (평가)
       ↑                                          │
       └────────── 불합격 시 재지시 ───────────────┘
```

---

## Phase 1: 디자인 리서치 (gemini-cli)

UI 태스크 수신 시, 구현 전에 gemini-cli로 디자인 방향을 먼저 잡는다.

```
gemini -p "
[Task] UI 디자인 제안
[Context] <현재 앱 상태, 기존 UI 구조 요약>
[User Request] <사용자가 원하는 UI 변경 내용>
[Expected Output]
  - 레이아웃 구조 (텍스트 와이어프레임 또는 설명)
  - 컴포넌트 목록
  - 인터랙션 흐름
  - 참고 사례 (있으면)
[Acceptance Criteria]
  - 사용자 요청의 모든 요소가 반영됨
  - 기존 UI와의 일관성 유지
"
```

**평가 기준:**
- 사용자 요청의 핵심 의도가 디자인에 반영되었는가
- 실현 가능한 수준인가 (현재 기술 스택 기준)

---

## Phase 2: 구현 (codex)

디자인 합격 후, codex에 구현 위임.

```
codex "
[Task] UI 구현: <한 줄 요약>
[Files] <대상 파일 목록>
[Design Spec]
  - <Phase 1에서 합격된 디자인 요약>
  - 레이아웃: <구조>
  - 컴포넌트: <목록>
  - 인터랙션: <흐름>
[Constraints]
  - 기존 코드 스타일 준수
  - 접근성 기본 요건 충족 (시맨틱 HTML, 키보드 내비게이션)
  - 반응형 고려 (필요 시)
[Acceptance Criteria]
  - 디자인 스펙의 모든 요소가 구현됨
  - 콘솔 에러 없음
  - 기존 기능 미파괴
" -y
```

---

## Phase 3: 검증 (claude-code)

codex 결과물을 평가. walkthrough.md 또는 출력 요약 기반.

| 판정 | 행동 |
|------|------|
| **합격** | 사용자에게 보고 |
| **디자인 문제** | Phase 1로 루프 (gemini 재지시) |
| **구현 문제** | Phase 2로 루프 (codex 재지시) |

---

## 복합 UI 태스크 분해 규칙

대규모 UI 변경은 아래 단위로 분해하여 각각 파이프라인을 돌린다:

1. **레이아웃/구조 변경** — 전체 골격
2. **컴포넌트 추가/수정** — 개별 UI 요소
3. **인터랙션/상태 연결** — 이벤트 핸들링, 상태 관리
4. **스타일링/마감** — 색상, 타이포그래피, 애니메이션

각 단위가 독립적으로 합격 판정을 받아야 다음 단계로 진행.

---

## 스크린샷 기반 검증 (선택)

구현 후 시각적 검증이 필요하면:

```
gemini -p "
[Task] UI 구현 결과 스크린샷 분석
[Input] <스크린샷 경로>
[Expected Output]
  - 디자인 스펙 대비 일치/불일치 항목
  - 시각적 이슈 (정렬, 여백, 색상 등)
[Acceptance Criteria]
  - 디자인 스펙과 90% 이상 일치
"
```
