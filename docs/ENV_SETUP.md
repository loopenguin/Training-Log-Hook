# 훈련일지 자동화 환경 변수 및 사용자 설정 가이드

이 문서는 훈련일지 자동화 GitHub Actions 워크플로가 정상적으로 동작하기 위해 필요한 **환경 변수(사용자 설정값)**의 구조와 획득 방법, 그리고 GitHub 저장소에 세팅하는 방법을 안내합니다.

## 1. 필요한 환경 변수 목록

워크플로 실행 시 아래의 6가지 환경 변수가 필요합니다.
절대 이 변수들의 실제 값을 소스 코드에 직접 하드코딩해서는 안 되며, 반드시 **GitHub Secrets**를 통해 주입해야 합니다.

| 변수명 (Key) | 용도 및 설명 |
|---|---|
| `TARGET_SITE_URL` | 접속할 목표 사이트의 URL 주소입니다. |
| `TARGET_SITE_ID` | 목표 사이트 로그인을 위한 사용자 아이디입니다. |
| `TARGET_SITE_PW` | 목표 사이트 로그인을 위한 비밀번호입니다. |
| `GOOGLE_SHEET_ID` | 데이터를 읽어올 구글 스프레드시트 문서의 고유 ID입니다. |
| `GOOGLE_SERVICE_ACCOUNT` | 구글 스프레드시트에 접근(API 호출)하기 위한 서비스 계정의 인증 키입니다. |
| `DISCORD_WEBHOOK_URL` | 작업 완료 후 최종 결과를 수신할 디스코드 채널의 웹훅 주소입니다. |

---

## 2. 변수별 세부 스펙 및 획득 방법

### 2.1 목표 사이트 접속 정보 (`TARGET_SITE_*`)
크롤링 대상이 되는 사이트의 정보입니다.
- **`TARGET_SITE_URL`**: 로그인 화면이나, 로그인 직후 이동할 데이터 갱신 페이지의 전체 URL을 적습니다. 
  - *(예: `https://www.example.com/admin/login`)*
- **`TARGET_SITE_ID`** & **`TARGET_SITE_PW`**: 해당 사이트에 실제로 로그인 가능한 계정 정보입니다. 보안을 위해 최소 권한만 부여된 크롤링 전용 더미(dummy) 계정을 새로 발급받아 사용하는 것을 권장합니다.

### 2.2 구글 스프레드시트 문서 ID (`GOOGLE_SHEET_ID`)
가져올 구글 스프레드시트 문서의 주소 표시줄에서 URL의 중앙에 있는 긴 문자열입니다.
- **획득 방법**: 
  - 예를 들어 URL이 `https://docs.google.com/spreadsheets/d/1BxiMVs0XRYFgwnTE_xxxx_yyyy/edit#gid=0` 라면,
  - `GOOGLE_SHEET_ID`는 **`1BxiMVs0XRYFgwnTE_xxxx_yyyy`** 가 됩니다.

### 2.3 구글 서비스 계정 인증키 (`GOOGLE_SERVICE_ACCOUNT`)
구글 시트는 비공개 문서인 경우가 대부분이므로, 봇이 이 문서에 접근하기 위해서는 '서비스 계정(Service Account)'이 필요합니다.
- **획득 방법**:
  1. [Google Cloud Console](https://console.cloud.google.com/) 에 접속합니다.
  2. 새 프로젝트를 생성하고, **Google Sheets API**를 활성화(Enable) 합니다.
  3. [IAM 및 관리자] > [서비스 계정] 으로 들어가 새 서비스 계정을 만듭니다. (예: `bot-sheet@my-project.iam.gserviceaccount.com`)
  4. 해당 서비스 계정의 **키(Key) 탭**에서 **새 키 만들기(JSON)** 를 클릭해 파일을 다운로드합니다.
  5. 다운로드한 `.json` 파일의 전체 내용을 텍스트 편집기로 열어서 복사한 뒤, 시크릿 변수에 넣습니다. (또는 줄바꿈 문제를 피하기 위해 전체를 Base64로 인코딩하여 넣기도 합니다. - *이 프로젝트의 구현 방식에 따름*)
  6. **중요:** 가져오려는 대상 스프레드시트 파일의 우측 상단 '공유' 버튼을 누르고, **이 서비스 계정의 이메일 주소(`bot-sheet@...`)에 뷰어 또는 편집자 권한**을 부여해야 데이터를 읽을 수 있습니다.

### 2.4 디스코드 웹훅 주소 (`DISCORD_WEBHOOK_URL`)
로직 처리가 끝나고 메시지를 보낼 디스코드 채널의 주소입니다.
- **획득 방법**:
  1. 디스코드에서 알림을 받을 채널의 **[편집(톱니바퀴)]** 버튼을 클릭합니다.
  2. **[연동(Integrations)]** > **[웹훅(Webhooks)]** > **[새 웹훅 만들기]** 를 클릭합니다.
  3. 봇의 이름과 프로필을 정하고 **[웹훅 URL 복사]** 버튼을 누릅니다.
  4. 복사된 `https://discord.com/api/webhooks/...` 주소를 그대로 사용합니다.

## 3. GitHub Secrets 설정 방법 (사용법)

준비된 6개의 변수 값을 실제 GitHub Actions가 읽을 수 있도록 세팅하는 과정입니다.

1. 이 저장소의 GitHub 페이지에 접속합니다.
2. 상단 탭에서 **[Settings]** 를 클릭합니다.
3. 좌측 사이드바 메뉴에서 **[Secrets and variables]** 아래의 **[Actions]** 를 클릭합니다.
4. 초록색 **[New repository secret]** 버튼을 클릭합니다.
5. **Name** 에는 위 표의 변수명 (예: `TARGET_SITE_PW`)을 넣고, **Secret** 에는 발급받은 실제 내용(비밀번호 문자열)을 넣고 **[Add secret]** 을 누릅니다.
6. 위 과정을 반복하여 6개의 변수를 모두 등록합니다.

> ⚠️ 한 번 등록된 Secret은 작성자 본인도 다시 열람할 수 없으므로, 등록 전에 필요한 키 파일이나 비밀번호는 로컬 안전한 곳(Ex: 1Password 등)에 보관하세요.

## 4. 운영 및 모니터링 가이드

### 4.1 자동화 워크플로 수동 켜기/끄기 (ON/OFF)
작업을 잠시 멈춰야 하거나 방학/휴일 등 일시적으로 자동화를 끄고 싶을 때 GitHub 자체 기능으로 쉽게 On/Off 할 수 있습니다.

- **워크플로 끄기 (OFF 단전):** 
  1. GitHub의 **[Actions]** 탭으로 이동합니다.
  2. 좌측 목록에서 끄고 싶은 워크플로를 선택합니다.
  3. 화면 우측 상단의 **`...` (더보기)** 아이콘을 누른 후 **[Disable workflow]** 를 클릭하면 추후 켤 때까지 내부 작업이 즉각 중지됩니다.
- **워크플로 다시 켜기 (ON):**
  1. 동일한 화면에 접속하면 나타나는 상단 경고 메시지에서 **[Enable workflow]** 버튼을 누르면 다시 스케줄에 맞춰 정상 가동됩니다.
- **수동 즉시 실행:** 
  1. 시간(17:30)을 기다리지 않고 당장 로직을 한 번 실행해보고 싶다면, 워크플로 메뉴 내의 **[Run workflow]** 버튼을 누릅니다.

### 4.2 설정값(사용자 변수) 변경 방법
해당 사이트의 비밀번호가 바뀌었거나 디스코드 수신 채널이 변경된 경우 쉽게 재설정할 수 있습니다.

1. GitHub의 **[Settings]** > **[Secrets and variables]** > **[Actions]** 메뉴로 다시 이동합니다.
2. Repository secrets 목록 중 변경할 변수의 우측 ✏️ **(연필 모양) 아이콘** (`Update`)을 누릅니다.
3. 새로운 값을 덮어쓰고 **[Update secret]** 을 누르면 다음 실행 시부터 즉시 적용됩니다.

### 4.3 오류 장애 파악 및 사용자 알림 체계
이 파이프라인은 사용자가 매일 모니터링하지 않아도 장애를 즉각 파악할 수 있는 안전망이 구축되어 있습니다.

- **작업 실패 발생 알림 체계:** 
  - 스크립트 내부 구조가 여러 구간(사이트 로그인, 갱신 버튼 클릭, 구글 시트 다운로드, 데이터 파싱 등)으로 분리되어 있습니다.
  - 문제가 발생하면 무작정 '오류 발생'이라고 뜨는 것이 아니라, 봇이 `DISCORD_WEBHOOK_URL`로 **"어디서 에러가 났는가 (예: [Login Step] 비밀번호 불일치)"**를 명확히 검출하여 긴급 메시지로 전송합니다. 이를 통해 사용자는 문제를 빠르게 진단할 수 있습니다.
- **레포지토리 자체의 변경/삭제/오류 파악 체계:** 
  - GitHub 정책에 따라, 레포지토리가 소유자에 의해 삭제되거나 권한 오류가 생기는 경우에는 GitHub 측 시스템에서 계정 이메일로 직통 경고 메일을 발송하므로 인프라 레벨의 단절 또한 파악할 수 있습니다.
  - 저장소에 60일 이상 아무런 활동(Commit 등)이 없으면 방치된 것으로 간주해 워크플로가 일시 정지될 수 있는데, 이 경우에도 사전에 GitHub이 계정 주인의 이메일로 알림을 보냅니다.
